# P0 credential exposure in git history

**Discovered:** 2026-04-23 during Phase 0/1A commit staging
**Severity:** P0 — Let's Encrypt cert private keys for *.quietly.its.me and *.quietly.online committed to the Master repo from the very first commit and updated in every subsequent commit.

## Files leaked

1. `.docker/traefik/acme.json` — tracked from initial commit `b38b0de` (Jan 2026) through `c150f84` (Feb 16). Contains CF DNS-01 Let's Encrypt account key + server certs with embedded private keys.
2. `.docker/traefik/config/config/acme.json` — same file under alternate path, also tracked historically.

## Action taken in this sprint

- `git rm --cached` on both paths — file content no longer changes in future commits
- `.gitignore` already contains `acme.json` — prevents re-addition
- **History is NOT rewritten** — doing so requires force-push to `quietly007/master` which is destructive and requires explicit user decision.

## Required follow-up (Phase 1F / Phase 5)

1. **Rotate Let's Encrypt certificates** — revoke old certs, delete `acme.json`, let Traefik re-issue via DNS-01. This is the ONLY way to invalidate the leaked private keys. Downtime window required.
2. **Rotate the Cloudflare DNS API token** — the token is in `~/.secrets/traefik/.env` (not in git, verify), but since acme.json was public, treat CF token as compromised too.
3. **Consider history rewrite** — `git filter-repo --invert-paths --path .docker/traefik/acme.json --path .docker/traefik/config/config/acme.json` + force-push to private GitHub repo. Only if repo access has ever been broadened beyond qui3tly personal account. Private repo → theoretical exposure only if GitHub itself breached or repo access widened.

## Related historical leak candidates

From `git log --all --diff-filter=A --name-only` (filtered for likely-secret paths):

- `.docs/howto/SECRETS.md` + `.docs/security/SECRETS.md` + `.docs/backup-20260122/*/SECRETS.md` — documentation that may have embedded actual secrets; file has since been deleted

These need to be walked in git history as part of Phase 1F-R-CRED-1-5.

## Immediate risk mitigation

Private GitHub repo with single-user access (qui3tly) — exposure is limited to:
- GitHub itself being compromised
- qui3tly GitHub account being compromised
- Any third party ever granted read access (none known)

Until certs are rotated, anyone with historical read access to the repo can MITM any `*.quietly.its.me` or `*.quietly.online` client that bypasses HSTS/HPKP.
